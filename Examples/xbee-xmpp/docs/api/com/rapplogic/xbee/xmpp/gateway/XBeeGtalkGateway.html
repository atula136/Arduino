XBee Gateway implementation for Google talk. As long as you have access to the Google Talk server, your XMPP gateway and clients can be on different networks, even behind firewalls.

This solution allows you to share your xbee hardware with anyone on the internet to experiment with as well as create XBee applications that reside on different physical machines and/or networks than the XBee. This is also a great way to distribute your XBee applications across the internet, without requiring a server or static ip address -- Google is the server. This is especially relevant to those with home networks that are not able to run servers, either by ISP policy or if they do not get a public ip address.

This solution allows your gateway can be brought down and upgraded without affecting the clients. When the gateway is brought up, clients will automatically get notified and resume communication. The client can go on and offline at will, and the gateway will always be notified if of the client's status.

You will need a minimum of two gtalk accounts: one for the gateway and one for the client.

By default, Google Talk accepts messages sent to offline users, and delivers the messages when the user signs on. However due to the real-time nature of XBee communication, the gateway will only communicate with online clients.

Caveat: Google may rate limit your account if you send too much traffic. BTW, I don't know what "too much" is.

isAvailable

protected boolean isAvailable(org.jivesoftware.smack.packet.Presence presence)
Description copied from class: XBeeXmpp
Subclass must implement to determine if the user (client or gateway) is online and able to receive packets. If this method returns true for a gateway, it means the gateway will delivery the packet to the client. For clients, this method must return true for the client to deliver the packet to the gateway.

Unfortunately the Presence object does not provide methods for away, available etc.; instead the presence string presence.toString(), must be evaluated.

The subclass should returns true only if the remote user is able to receive messages.

The presence string varies depending on client sending the presence.
Google Talk uses "available ()" and the "dnd" token to represent busy (e.g. "available: dnd ()" for busy)
Google Talk will send "available: away ()" when it thinks the computer is idle
Google Talk custom status are placed within parentheses (e.g. "available (Grrrr)")
iChat seems to send "available" or "available (At home)" where the custom status is in parentheses
iChat sends "available: xa" if user is online but away
Both Google Talk and iChat send "unavailable" when a user signs out.
So to summarize:
If presence contains "available: dnd" or available: away, Google Talk user is online but away/busy.
I believe "dnd" indicates the user set the status, where the "away" status is set by Google Talk
If presence contains "available: xa", iChat user is online but away/busy
If presence contains unavailable, user is not online
Otherwise user is considered online and available
